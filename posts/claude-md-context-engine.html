CLAUDE.md：把 Claude 从“临时聊天”变成“长期工作伙伴”

多数人使用 Claude 的方式，其实还停留在“每次重新认识一次”的阶段。

你打开一个新会话，它不知道你是谁，不知道你正在做什么，不知道你偏好的表达方式，也不知道你有哪些不能被破坏的规则。于是你不得不反复解释：背景、目标、语气、约束、技术栈、哪些能改、哪些不能碰。

这不是模型不聪明，而是上下文没有被制度化。

CLAUDE.md 的价值就在这里：它不是一个神秘技巧，而是把“你每次都要重复说的话”沉淀成项目级、可复用、可持续迭代的协作协议。

flowchart TD
    A["打开新的 Claude 会话"] --> B{有没有 CLAUDE.md?}
    B -->|没有| C["重新解释背景、偏好、边界"]
    C --> D["输出容易偏离你的工作方式"]
    B -->|有| E["自动读取项目规则与上下文"]
    E --> F["直接进入高质量协作"]
    F --> G["持续把经验沉淀回文件"]

一、CLAUDE.md 解决的不是“提示词”，而是“连续性”

普通 Prompt 是一次性的。你说完，模型执行；会话结束，很多细节就消失了。

CLAUDE.md 不一样。它更像项目里的“协作宪法”：

  • 你是谁；
  • 当前项目在做什么；
  • 输出要遵守什么风格；
  • 哪些行为必须先问；
  • 哪些文件不能随便改；
  • 遇到不确定事实时应该如何处理；
  • 做完任务后应该如何汇报变化。

这让 Claude 不再只是一个“能回答问题的聊天窗口”，而更像一个被你训练过协作方式的长期助手。

graph LR
    P["一次性 Prompt"] --> P1["只影响当前问题"]
    P1 --> P2["下次仍要重复说明"]

    M["CLAUDE.md"] --> M1["项目级持久规则"]
    M1 --> M2["每次会话自动生效"]
    M2 --> M3["越用越贴合你的工作流"]

二、它不是开发者专属，任何严肃使用 AI 的人都需要

很多人听到 CLAUDE.md，会以为这是 Claude Code 的工程配置文件。确实，它在开发场景里非常强，但它的本质并不局限于代码。

  • 写作者可以用它锁定个人文风，避免 Claude 写出“AI 味”。
  • 研究者可以用它固定文献整理结构、引用习惯和不确定性标注方式。
  • 市场/运营人员可以用它定义受众、语气、禁用词和品牌边界。
  • 创业者/管理者可以用它写入公司背景、产品定位、决策原则。
  • 开发者可以用它约束技术栈、修改范围、测试要求和危险操作边界。

换句话说，只要你不是偶尔问两句天气，而是想让 AI 参与长期工作，CLAUDE.md 就值得尽早建立。

三、最小可用版本：两分钟就够

你不需要一开始就写一份完美的 300 行配置。真正有效的方式是：先写 3-5 条最常用、最容易减少返工的规则。

创建方式很简单：

  1. 在项目根目录新建一个文件：CLAUDE.md。
  2. 文件名必须准确：大写 CLAUDE，扩展名 .md。
  3. 写入你希望 Claude 每次都遵守的规则。
  4. 保存后，在该目录开启 Claude 会话，它会自动读取。

一个最小版本可以这样开始：

# CLAUDE.md

## 沟通风格
- 不要用“Great question”“当然可以”这类开场白，直接给结论。
- 简单问题简短回答，复杂任务再展开。
- 不确定的事实必须标注，不要编造。

## 工作边界
- 不要修改我没有要求修改的内容。
- 删除文件、部署、发送外部消息前必须先确认。
- 完成代码任务后说明改了哪些文件、还有什么风险。

## 项目上下文
- 当前项目目标：填写你的项目目标。
- 主要用户：填写目标受众。
- 技术栈：填写语言、框架、包管理器、数据库等。

重点不是一口气写全，而是先覆盖你最痛的返工点。

四、21 条规则可以分成 5 类

原文列了 21 条指令。不要把它当成死记硬背的清单，更合理的方式是把它们归为五类：沟通、行为边界、个人/项目上下文、记忆机制、开发约束。

flowchart TB
    A["CLAUDE.md 规则体系"] --> B["沟通方式"]
    A --> C["行为边界"]
    A --> D["个人与项目上下文"]
    A --> E["记忆与复盘"]
    A --> F["开发工程约束"]

    B --> B1["去掉废话开场"]
    B --> B2["长度匹配任务复杂度"]
    B --> B3["不确定就明说"]

    C --> C1["大改前先问"]
    C --> C2["只改被要求的内容"]
    C --> C3["外部动作必须确认"]

    D --> D1["我是谁"]
    D --> D2["项目目标"]
    D --> D3["写作声音"]

    E --> E1["维护 MEMORY.md"]
    E --> E2["记录失败经验"]
    E --> E3["沉淀长期事实"]

    F --> F1["锁定技术栈"]
    F --> F2["最简单方案优先"]
    F --> F3["不碰无关代码"]

下面逐类拆开。

五、沟通规则：先让 Claude 别浪费你的注意力

最值得优先写进 CLAUDE.md 的，是沟通方式。

很多 AI 回复的问题不在于错，而在于“噪音太多”：

  • “Great question!”
  • “Of course!”
  • “Certainly!”
  • 结尾再重复一遍刚说过的话；
  • 简单问题写四段，复杂问题却只给骨架。

这些东西单次看没什么，但每天用几十次就会变成注意力税。

推荐写入：

## Response Style
- Never open with filler phrases like “Great question”, “Of course”, or “Certainly”. Start with the answer.
- Match response length to task complexity. Simple questions get short answers; complex tasks get structured detail.
- If a fact, quote, date, or statistic is uncertain, say so explicitly before using it.

这类规则的收益非常稳定：它不会让 Claude 突然变聪明，但会让每一次交互更干净。

六、行为边界：阻止 AI “好心办坏事”

AI 最大的麻烦之一，是它经常把“完成任务”理解成“顺手优化一切”。

你让它改一个段落，它重写全文；你让它修一个 bug，它顺便重构三个文件；你让它整理内容，它删掉了你其实想保留的部分。

所以 CLAUDE.md 里必须有边界规则。

sequenceDiagram
    participant U as 用户
    participant C as Claude
    participant F as 文件/外部系统

    U->>C: 修改这一段 / 修这个 bug
    C->>C: 判断是否会显著改变已有内容
    alt 只影响指定范围
        C->>F: 执行修改
        C->>U: 汇报改动
    else 会大改、删除、部署或外发
        C->>U: 说明影响范围并请求确认
        U->>C: 明确确认后再继续
    end

建议写入：

## Boundaries
- Only change what I explicitly asked you to change.
- Before significantly altering existing content, explain the change and wait for confirmation.
- Never send, publish, deploy, schedule, delete files, or run irreversible actions without explicit confirmation in the current conversation.
- After editing or coding tasks, summarize what changed, what stayed untouched, and what needs attention.

这部分非常重要。因为越是强大的工具，越需要明确的刹车。

七、上下文规则：让输出更像“为你写的”

Claude 默认不知道你的背景。它不知道你是新手还是专家，不知道你喜欢技术深挖还是快速结论，也不知道你的读者是谁。

于是它只能猜。猜对了像默契，猜错了就是返工。

你可以在 CLAUDE.md 里写入稳定背景：

## About Me
- Name: 填写你的名字或昵称。
- Role: 填写你的身份，例如研究生、开发者、独立创作者。
- Strong in: 填写你熟悉的领域。
- Still learning: 填写你需要更多解释的领域。
- Preferred depth: 对熟悉领域不要过度解释，对陌生领域给必要背景。

## Project Context
- Project: 当前项目名称。
- Goal: 项目目标。
- Audience: 目标用户或读者。
- Tone: 输出语气。
- Avoid: 明确不要出现的表达、方案或风格。

对于写作类任务，还可以单独加一块：

## Writing Voice
- Use concise, concrete Chinese.
- Avoid generic AI-style transitions.
- Prefer clear structure, examples, and practical judgment.
- Do not use exaggerated marketing language unless asked.

这会显著减少“内容看起来正确，但不像你会发出来的东西”。

八、记忆规则：用文件弥补会话遗忘

Claude 的会话可能会结束，聊天上下文可能会丢，但文件可以留下。

因此，CLAUDE.md 可以规定一些长期记忆机制，例如：

  • 重要决策写入 MEMORY.md；
  • 失败尝试写入 ERRORS.md；
  • 每次结束会话时写总结；
  • 稳定事实单独维护，避免下次重新解释。
graph TD
    A["一次工作会话"] --> B{发生重要信息?}
    B -->|决策| C["写入 MEMORY.md"]
    B -->|踩坑| D["写入 ERRORS.md"]
    B -->|长期事实| E["写入 STABLE_FACTS.md 或 CLAUDE.md"]
    B -->|无复用价值| F["不记录，避免噪音"]

    C --> G["下次会话读取"]
    D --> G
    E --> G
    G --> H["减少重复解释与重复踩坑"]

示例规则：

## Memory
- Maintain MEMORY.md for important decisions: what was decided, why, and what alternatives were rejected.
- Maintain ERRORS.md when an approach takes more than two attempts: what failed, what worked, and what to remember next time.
- When I say “session end”, write a short session summary: completed, in progress, decisions, and next steps.
- Do not store temporary task progress as permanent memory unless it will matter in future sessions.

这里要注意：记忆不是越多越好。垃圾记忆会污染未来判断。真正应该沉淀的是稳定偏好、项目约束、踩坑经验和关键决策。

九、开发规则：让 Claude Code 像工程师，而不是热心实习生

在代码场景里，CLAUDE.md 的价值会被进一步放大。

因为代码任务里的错误常常不是“没写出来”，而是：

  • 改了无关文件；
  • 引入没必要的抽象；
  • 没跑测试；
  • 默认换技术栈；
  • 对不确定需求直接猜；
  • 修改范围太大，review 成本爆炸。

建议把开发规则写得非常具体：

## Engineering Rules
- Ask when requirements are ambiguous. Do not make silent assumptions.
- Use the simplest solution that could work.
- Do not refactor, rename, or reorganize unrelated code.
- Only modify files directly related to the current task.
- Use the existing tech stack and package manager unless I explicitly ask for alternatives.
- Before deleting files, running migrations, deploying, or calling irreversible external services, ask for confirmation.
- After coding tasks, report: files changed, what was modified, files not touched, tests run, follow-up needed.

这几条看起来朴素，但很有效。它们把 Claude 从“积极但不可控”拉回到“可审查、可回滚、可协作”。

flowchart LR
    A["开发任务"] --> B["先读现有代码"]
    B --> C{需求是否明确?}
    C -->|否| D["提问澄清"]
    C -->|是| E["最小修改集"]
    E --> F["运行测试/校验"]
    F --> G["汇报文件变化与风险"]

    E -.禁止.-> H["无关重构"]
    E -.禁止.-> I["擅自换栈"]
    E -.禁止.-> J["未确认部署"]

十、推荐的 CLAUDE.md 模板

下面是一份可以直接复制的通用模板。你可以先用它，再按项目慢慢增删。

# CLAUDE.md

## Role
You are my project co-pilot. Be concise, careful, and practical.

## Response Style
- Start with the answer. No filler openings.
- Match length to task complexity.
- Flag uncertainty explicitly. Do not invent facts, dates, quotes, or statistics.
- Prefer concrete examples over vague advice.

## Boundaries
- Only change what I asked you to change.
- Before large rewrites, destructive actions, deployment, publishing, scheduling, or external messages, ask for explicit confirmation.
- If you notice unrelated improvements, mention them separately. Do not apply them without permission.

## Context
- About me: [name, role, background]
- Project: [project name]
- Goal: [project goal]
- Audience: [target users/readers]
- Tone: [preferred tone]
- Avoid: [things to avoid]

## Writing Voice
- [Describe your writing style]
- [Words or patterns to avoid]
- [Formatting preferences]

## Engineering Rules
- Ask when ambiguous.
- Simplest solution first.
- Do not touch unrelated files.
- Use the existing stack and package manager.
- Run relevant tests or explain why tests were not run.
- End with files changed, tests run, and follow-up needed.

## Memory
- Record important decisions in MEMORY.md.
- Record repeated failures or tricky fixes in ERRORS.md.
- Keep memory compact and reusable.

如果只想从 3 条开始，我会选这三条：

- Start with the answer. No filler openings.
- Only change what I explicitly asked you to change.
- If uncertain, say so. Never fill gaps with plausible-sounding facts.

因为它们分别解决了三个最常见问题：废话、越界、幻觉。

十一、如何让它长期有效：把 CLAUDE.md 当成活文档

CLAUDE.md 不应该写完就不动。它应该像项目 README、架构决策记录一样持续演化。

每次你发现 Claude 反复犯同一种错，就把对应规则写进去。

flowchart TD
    A["发现一次反复出现的问题"] --> B["判断是否具有长期复用价值"]
    B -->|没有| C["本次纠正即可"]
    B -->|有| D["写入 CLAUDE.md / MEMORY.md / ERRORS.md"]
    D --> E["下次会话自动应用"]
    E --> F["观察是否减少返工"]
    F -->|仍然不够| G["把规则写得更具体"]
    G --> E

好的规则通常有几个特征：

  • 具体：不要写“写得好一点”，要写“不要使用夸张营销语”。
  • 可执行：不要写“保持谨慎”，要写“外部发送前必须确认”。
  • 稳定：不要把一次性任务进度塞进去。
  • 可验证：最好能在任务结束时检查是否遵守。

十二、我的判断：CLAUDE.md 的核心不是“让 AI 听话”，而是降低协作摩擦

原文把 CLAUDE.md 说得很有冲击力：它能让 Claude 从零记忆变成带上下文协作。我基本同意，但要补一句：

CLAUDE.md 不会替你完成思考，也不会保证模型永远不犯错。它真正稳定提升的是三个东西：

  1. 减少重复解释：固定背景和偏好不用每次重说。
  2. 减少越界操作：大改、删除、发布、部署都有刹车。
  3. 减少风格偏移：输出更接近你的表达和项目需求。

如果你把它当成万能 Prompt，会失望；如果你把它当成“协作协议”，它会非常值。

十三、结语：先写三条，再让它长出来

不要等到你有一份完美的 CLAUDE.md 才开始。最好的做法是今天就建一个最小版本：

  • 去掉废话；
  • 不确定就说；
  • 不要擅自大改；
  • 写入你的身份和项目背景；
  • 对开发任务加上修改范围和测试要求。

然后每次返工后问一句：这条经验值得沉淀吗？

如果值得，就写进文件。

长期来看，真正改变效率的不是某一次神奇 Prompt，而是你把协作规则一点点变成系统。CLAUDE.md 就是这个系统最简单的入口。
